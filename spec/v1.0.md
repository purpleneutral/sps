## 1. Purpose

The Seglamater Privacy Specification (SPS) defines a set of measurable, externally observable checks that evaluate the privacy posture of a publicly accessible website. Each check is scored; the total yields a score from 0 to 100 and a letter grade.

All checks are based on data observable without authentication or cooperation from the site operator: HTTP headers, DNS records, TLS handshake parameters, and page content.

---

## 2. Scoring

The specification defines **six categories** totaling **100 points**:

| Category | Max Points |
|----------|-----------|
| Transport Security | 20 |
| Security Headers | 20 |
| Tracking & Third Parties | 30 |
| Cookie Behavior | 15 |
| Email & DNS Security | 10 |
| Best Practices | 5 |

### 2.1 Grade Thresholds

| Grade | Score Range | Badge Color |
|-------|------------|-------------|
| A+ | 95–100 | Bright green |
| A | 90–94 | Green |
| B | 75–89 | Blue |
| C | 60–74 | Yellow |
| D | 40–59 | Orange |
| F | 0–39 | Red |

Sites scoring **A or A+** are eligible for the "Seglamater Verified" badge variant.

## 3. Transport Security (20 points)

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `tls_1_3_supported` | TLS 1.3 support | 8 | Server successfully negotiates a TLS 1.3 handshake |
| `tls_legacy_disabled` | TLS 1.0/1.1 disabled | 4 | Server does not accept TLS 1.0 or TLS 1.1 connections |
| `hsts_enabled` | HSTS enabled | 4 | `Strict-Transport-Security` response header is present |
| `hsts_max_age` | HSTS max-age >= 1 year | 2 | HSTS `max-age` directive is at least 31,536,000 seconds |
| `hsts_subdomains` | HSTS includeSubDomains | 1 | HSTS header includes the `includeSubDomains` directive |
| `hsts_preload` | HSTS preload | 1 | HSTS header includes the `preload` directive |

### Method

- Initiate TLS handshakes specifying TLS 1.3 and TLS 1.2 protocol versions
- Read the `Strict-Transport-Security` response header from an HTTPS GET request to the domain root
- Parse HSTS directives (max-age, includeSubDomains, preload)

## 4. Security Headers (20 points)

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `csp_present` | Content-Security-Policy present | 6 | `Content-Security-Policy` response header exists |
| `csp_no_unsafe_inline` | CSP blocks unsafe-inline | 3 | `script-src` (or `default-src`) does not contain `'unsafe-inline'` |
| `csp_no_unsafe_eval` | CSP blocks unsafe-eval | 3 | `script-src` (or `default-src`) does not contain `'unsafe-eval'` |
| `referrer_policy` | Referrer-Policy set | 3 | `Referrer-Policy` header is present with a restrictive value: `no-referrer`, `same-origin`, `strict-origin`, or `strict-origin-when-cross-origin` |
| `permissions_policy` | Permissions-Policy set | 3 | `Permissions-Policy` header restricts at least one sensitive API: camera, microphone, geolocation, accelerometer, gyroscope, magnetometer, payment, usb, bluetooth |
| `x_content_type_options` | X-Content-Type-Options | 1 | Set to `nosniff` |
| `x_frame_options` | X-Frame-Options | 1 | Set to `DENY` or `SAMEORIGIN` |

### Method

- Make an HTTPS GET request to the domain root
- Parse each response header according to its specification

## 5. Tracking & Third Parties (30 points)

This is the highest-weighted category because it has the most direct impact on user privacy.

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `no_analytics` | No third-party analytics scripts | 10 | No `<script>` tags load resources from known analytics domains |
| `no_trackers` | No advertising/tracking scripts | 10 | No resources loaded from known advertising or tracking domains |
| `no_fingerprinting` | No fingerprinting patterns | 5 | No inline scripts contain known fingerprinting API calls or library signatures |
| `no_third_party_cdns` | No third-party fonts/CDNs | 3 | All resources (scripts, stylesheets, fonts, images) are served from the first-party domain |
| `all_https` | All resources over HTTPS | 2 | No resources loaded over HTTP (mixed content) |

### Method

- Fetch the page HTML and parse all `<script>`, `<link>`, `<img>`, and `<iframe>` elements
- Resolve relative URLs to absolute using the page URL as base
- Compare each resource domain against built-in and vendored blocklists:
  - Known analytics domains (Google Analytics, Hotjar, Mixpanel, etc.)
  - Known tracker/ad domains (DoubleClick, Facebook Pixel, etc.)
  - Known CDN domains (fonts.googleapis.com, cdnjs.cloudflare.com, etc.)
- Scan inline scripts for fingerprinting patterns:
  - Canvas: `toDataURL`, `getImageData`
  - WebGL: `UNMASKED_VENDOR_WEBGL`, `UNMASKED_RENDERER_WEBGL`
  - AudioContext: `OfflineAudioContext`, `createOscillator`
  - Libraries: FingerprintJS signatures
  - Hardware: `navigator.hardwareConcurrency` + `navigator.deviceMemory` enumeration

### Blocklist Sources

- Built-in curated list of known analytics, tracker, and CDN domains
- External lists (vendored and periodically updated):
  - [EasyPrivacy](https://easylist.to/easylist/easyprivacy.txt)
  - [Disconnect](https://github.com/nicediscuss/nicediscuss-lists)
  - [Peter Lowe's ad server list](https://pgl.yoyo.org/adservers/)

## 6. Cookie Behavior (15 points)

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `no_third_party_cookies` | No third-party cookies | 5 | No `Set-Cookie` headers from third-party domains |
| `cookies_secure` | All cookies have Secure flag | 3 | Every cookie includes the `Secure` attribute |
| `cookies_httponly` | All cookies have HttpOnly flag | 3 | Every cookie includes the `HttpOnly` attribute |
| `cookies_samesite` | All cookies have SameSite attribute | 2 | Every cookie includes `SameSite=Strict` or `SameSite=Lax` |
| `cookies_expiration` | Reasonable cookie expiration | 2 | No cookie has a `Max-Age` or `Expires` exceeding 1 year (31,536,000 seconds) |

### Method

- Collect all `Set-Cookie` response headers from the initial page load
- Parse each cookie's attributes: `Secure`, `HttpOnly`, `SameSite`, `Max-Age`, `Expires`, `Domain`
- A cookie is third-party if its `Domain` attribute does not match the scanned domain

### Note

If no cookies are set, all checks in this category pass (a site that sets no cookies exhibits ideal cookie behavior).

## 7. Email & DNS Security (10 points)

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `spf_present` | SPF record present and strict | 3 | TXT record containing `v=spf1` exists and ends with `-all` (hard fail) |
| `dkim_present` | DKIM record discoverable | 2 | A DKIM TXT record is found via common selector probing (`default`, `google`, `selector1`, `selector2`, `k1`, `dkim`, `mail`, `s1`, `s2`, etc.) |
| `dmarc_policy` | DMARC policy set | 3 | `_dmarc` TXT record exists with `p=quarantine` or `p=reject` |
| `dnssec_enabled` | DNSSEC enabled | 1 | DNSKEY records are present for the domain |
| `caa_present` | CAA record present | 1 | At least one CAA DNS record exists |

### Method

- Perform DNS TXT lookups for SPF, DKIM (via common selectors), and DMARC (`_dmarc.domain`)
- Perform DNS DNSKEY and CAA record lookups
- All DNS queries use standard protocol-level operations

## 8. Best Practices (5 points)

| ID | Check | Points | Criteria |
|----|-------|--------|----------|
| `security_txt` | security.txt present | 2 | `/.well-known/security.txt` returns HTTP 200 with valid content per RFC 9116 |
| `privacy_json` | privacy.json present | 2 | `/.well-known/privacy.json` returns HTTP 200 with valid JSON |
| `accessible_without_js` | Accessible without JavaScript | 1 | The initial HTML response contains meaningful text content (at least 20 words after stripping tags, and does not consist solely of SPA bootstrap code) |

### Method

- HTTPS GET requests to well-known paths
- For security.txt: validate presence of required `Contact:` field and recommended `Expires:` field
- For privacy.json: validate that the response parses as valid JSON
- For JS-free accessibility: strip HTML tags from the response body and count words; detect SPA indicators (`<div id="root"></div>`, "please enable javascript", etc.)

## 9. The privacy.json Proposal

SPS v1.0 proposes a machine-readable privacy declaration at `/.well-known/privacy.json`:

```json
{
  "spec": "seglamater-privacy-spec",
  "version": "1.0",
  "claims": {
    "analytics": "none",
    "third_party_scripts": "none",
    "cookie_policy": "essential-only",
    "data_retention_days": 30,
    "ip_logging": false,
    "user_tracking": false,
    "data_sharing": "none"
  },
  "contact": "privacy@example.com",
  "last_reviewed": "2026-02-25"
}
```

Claims that can be verified externally (analytics, third-party scripts, cookies) are cross-checked by the scanner. Claims that cannot be verified externally (data retention, IP logging) are displayed on the verification page as **"self-reported, unverified."**

## 10. Versioning

- The specification uses **semantic versioning** (MAJOR.MINOR)
- **MINOR** versions add optional bonus checks or refine scoring weights
- **MAJOR** versions may add or remove required checks, or change grade thresholds
- Scan results always reference the specification version used
- Old specification versions remain documented for historical comparison
- Community proposals are accepted via pull requests to the specification repository
